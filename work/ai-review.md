# 자격증봄 교차 검토

## 제품·UX 검토

- 문제와 근거. 시험 서비스는 큰 마케팅 문구보다 지금 접수할 수 있는지와 다음 행동이 먼저 보여야 한다. 홈 상단에 검색과 `나에게 맞는 시험 찾기`를 배치하고, 확인된 다음 일정만 이어서 표시했다.
- 사용자 영향. 40·50대 사용자가 로그인 없이 검색하고 두 번 이내 공식 접수처로 이동할 수 있다.
- 모바일 영향. 모든 주요 터치 영역을 최소 48px로 두고, 네 개의 하단 메뉴와 safe area를 적용했다.
- 난이도와 위험. 중간 난이도, 기존 앱이 없어 회귀 위험은 낮다. 정보가 많아지는 위험을 상세 화면의 단계적 공개로 줄였다.
- 결정. 채택했다. 거대한 히어로와 과도한 배지는 제외했다.

## 공식 데이터 검토

- 문제와 근거. 공식 기관별 일정 구조와 시각 정밀도가 달라 한 가지 D-day 모델로 합치면 거짓 알림이 생긴다.
- 사용자 영향. 상시 시험이나 날짜만 있는 공고에서 잘못된 마감 시각을 믿는 위험이 크다.
- 모바일 영향. 일정 카드에 `정확 시각`, `표준 시각 기준`, `날짜만 확인`을 쉬운 문구로 드러내야 한다.
- 난이도와 위험. 높은 난이도다. Q-Net API만 자동 어댑터로 시작하고 나머지는 검증된 수동 데이터 또는 공식 링크로 제한했다.
- 결정. 채택했다. HTML 자동 수집과 샘플 운영 날짜는 제외했다.

## 보안·운영 검토

- 문제와 근거. 로그인·푸시가 붙으면 관심 시험, endpoint, 운영 원본을 같은 권한으로 노출할 수 없다.
- 사용자 영향. 개인 일정과 푸시 기기 정보 유출을 막아야 한다.
- 모바일 영향. 키가 없어도 로그인 없는 탐색과 로컬 관심 저장은 중단되지 않아야 한다.
- 난이도와 위험. 높은 난이도다. RLS, 명시적 grants, endpoint 암호문 저장 계약, 관리자 app metadata 정책을 migration에 고정했다.
- 결정. 스키마와 비활성 UI는 채택했다. 자격 증명 없이 OAuth·Web Push가 동작한다고 꾸미는 것은 제외했다.

## 리드 최종 범위

이번 배포는 0.4.0으로 제한한다. 공식 시험 10개 탐색, 설명 가능한 추천 3+7, 확인된 일정, 공식 준비물, 관심 저장, 공유·캘린더, PWA를 실제 배포한다. Supabase·OAuth·Web Push·자동 동기화는 안전한 코드 계약과 운영 문서까지만 포함하며 외부 키 등록 전에는 활성화하지 않는다. 이 범위가 거짓 데이터 없이 즉시 쓸 수 있는 가장 큰 배포 단위다.

## 0.5.0 카탈로그 확장 회의

### 데이터 담당 독립 검토

- 문제와 근거. `packages/core/src/catalog.ts`는 시험 10개뿐이고 한국사 외에는 대부분 일정이 비어 있었다. Vercel과 GitHub 환경에도 `QNET_SERVICE_KEY`가 없어서 실제 API 호출은 없지만 일부 Q-Net 카드가 `official-api`로 표시됐다.
- 사용자 영향. 자격증을 검색해도 없거나 같은 `다음 회차 확인` 문구만 반복돼 서비스가 고장 난 것처럼 보였다.
- 모바일 영향. 날짜 없는 카드가 길게 반복돼 스크롤 비용만 늘고 지금 접수할 시험을 찾기 어려웠다.
- 난이도와 위험. 공식 기관마다 날짜 정밀도와 회차 구조가 달라 중간 이상이다. 공유 일정을 잘못 복제하거나 추정 시각을 넣는 회귀 위험이 있다.
- 의견. 공식 API 표시는 제거하고, 공식 공고·PDF·일정 페이지의 검증 스냅샷으로 우선 확장한다. 상시검정은 날짜를 만들지 않는다. 채택했다.

### 모바일 UX 담당 독립 검토

- 문제와 근거. 기존 홈의 큰 추천 문구가 먼저 나오고 실제 보유 시험 수·접수 상태는 드러나지 않았다. 직접 찾기 화면은 10개를 모두 렌더링해 확장 후에는 과도한 스크롤이 예상됐다.
- 사용자 영향. 사용자가 원하는 `지금 신청할 수 있는 시험` 판단이 늦고 데이터가 늘수록 탐색이 더 어려워진다.
- 모바일 영향. 320×568 첫 화면에 직접 검색 CTA를 유지하고, 검색창 58px·필터 46px·주요 버튼 48px 이상을 확보해야 한다.
- 난이도와 위험. 낮음에서 중간이다. 고정 하단 메뉴와 더 보기 상태가 겹치지 않는지 세 화면 크기로 확인해야 한다.
- 의견. 홈에 전체·일정 보유·현재 접수 지표와 검색 CTA를 올리고, 직접 찾기는 접수 우선 정렬과 12개씩 더 보기를 사용한다. 채택했다.

### 접근성·안정성 담당 독립 검토

- 문제와 근거. 모든 일반 시험의 준비물 ID가 `identity-check`로 같아 한 시험을 체크하면 다른 시험도 체크된 것으로 보였다. 날짜 범위 ICS도 종료일을 무시하고 하루 일정으로 저장했다.
- 사용자 영향. 준비 완료 상태와 개인 일정이 잘못 저장될 수 있다.
- 모바일 영향. 작은 화면에서 잘못된 체크 상태를 다시 찾아 고치는 부담이 크다.
- 난이도와 위험. 낮지만 데이터 무결성에 직접 영향을 준다.
- 의견. 시험 ID를 준비물 ID에 포함하고 날짜 범위의 종료 다음 날을 ICS 종일 종료일로 사용한다. 채택했다.

### 성능 담당 독립 검토

- 문제와 근거. 97개 정적 시험을 한 번에 번들에 넣으면 초기 자바스크립트가 늘지만, 직접 찾기에서는 12개만 렌더링한다.
- 사용자 영향. 저사양 모바일에서 초기 로딩이 느려질 가능성이 있다.
- 모바일 영향. 렌더링 카드 수 제한은 스크롤과 레이아웃 비용을 함께 줄인다.
- 난이도와 위험. 별도 청크·지연 로딩은 구조와 오프라인 PWA 캐시를 복잡하게 만든다.
- 의견. 빌드 측정 전에는 분할하지 않고 gzip 증가가 큰 경우만 재검토한다. 실제 증가는 7.19KB여서 지연 로딩은 제외했다.

### Claude Code 의견 교환

- 전달 내용. 시험 부족, API 오인 표시, 공식 일정 확대 후보, 접수 우선 정렬, 공통 회차 중복, 모바일 더 보기, 정적 데이터 번들 위험을 전달했다.
- Claude Code 의견. API 표기 수정, 별칭 정규화, 마지막 검증일, 추천 3+7 회귀 테스트, 공통 일정 묶음, 48px 터치 영역은 필요하다고 했다. 60개 이상 정적 데이터는 지연 로딩을 검토하고, 날짜 미확정 시험을 많이 넣으면 긴급성이 희석될 수 있다고 경고했다.
- 채택. API 표기 수정, 별칭 검색, 검증일, 공통 일정 묶음, 3+7 테스트, 48px 이상 터치 영역을 채택했다.
- 보완. 날짜 없는 시험은 `상시 접수`와 `공고 확인형`으로 분리하고, 일정 보유 70개와 현재 접수 수를 상단에서 따로 드러냈다.
- 반박. 일반 객체 97개의 측정 증가가 gzip 약 7.2KB에 그쳐 지연 로딩은 효과보다 복잡성이 크다. 12개씩 렌더링하는 것으로 충분하다고 판단했다.

### 교차 검토와 리드 최종 결정

- 데이터 담당은 UX의 대량 확장안을 검토해 공통 회차 21개가 홈을 점유하는 문제를 지적했고, `groupKey` 묶음을 추가했다.
- UX 담당은 데이터 담당의 보수적 날짜 정책을 검토해 `API 미연결` 반복 대신 상시 시험의 실제 다음 행동인 시험장·날짜 선택 문구를 제안했다.
- 안정성 담당은 더 보기와 추천이 기존 3+7 흐름을 깨지 않는지 320·390·1280px 브라우저 테스트를 요구했다.
- 최종 범위. 시험 97개, 공식 출처 8개, 일정 보유 시험 70개, 접수 우선 정렬, 별칭 검색, 상태·분야 필터, 12개씩 더 보기, 범위 날짜, 출처·검증일, 준비물 ID 수정만 채택했다. 자동 HTML 수집, 키 없는 API 활성화, 근거 없는 날짜, 측정 효과가 작은 지연 로딩은 제외했다.
